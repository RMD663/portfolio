Slime Dungeon

Battle arena game with swarms of enemies on the screen. And stackable power ups.

About

Slime Dungeon is a 3D arena battle game developed for Mini Jam 202 on Itch.io. Developed solo in just 72 hours, the project served as a technical exercise focused on managing high-density entity counts and building a decoupled, scalable combat architecture using the Godot Resource system.

I was solely responsible for the project, with a primary focus on designing and implementing the combat system and the state managers that drive the internal game logic.

Main Challenge

The primary technical challenge was maintaining a stable 60 FPS on Web builds while simulating hundreds of simultaneous enemies and physics-based interactions.

Mechanics

As the solo developer, I was responsible for:

  • Implementation of a Node Based FSM: Implemented a Finite State Machine leveraging Godot Node structure, allowing for modular behavior.
  • Enemy AI: Developing a state-based AI system, that allows to create and reuse behaviors across different enemy types easily.
  • Player Movement System: Created custom "Slide" movement system designed to simulate slime-like physics and momentum.
  • Game Animations: Used Godot AnimationPlayer to implement squash-and-stretch techniques, adding visual feedback and responsiveness to player actions.
  • Performance Optimization: Profiled and optimized the game to ensure a stable framerate even with hundreds of active entities on screen.

Inovative Solutions

Data-Driven Damage System: To avoid static and repetitive combat, I implemented a data-driven damage system using Godot Resources. A custom DamageData resource stores damage values, direction, and knockback force, allowing each entity to independently decide how to process incoming impacts.

Modular Architecture: By adopting an FSM-based approach, I avoided colossal singular scripts. Logic was split into small, dedicated classes, making the project easier to maintain and expand under tight deadlines.

Challenges And Solutions

During the development, I found myself dealing with the following challenges:

1. Performance: Early builds of the game suffered from lag spikes due to simultaneous pathfinding requests and particle effects. This was solved by implementing a staggered update system that distributes expensive calculations across multiple frames, reducing CPU bottlenecks.

2. Enemy Collision: With hundreds of entities crammed together, they often became stuck. I addressed this by configuring custom physics layers and masks, allowing enemies to ignore collisions with each other while still interacting with the player and environment.

3. Game Feel: With high action density, visual clarity was critical. I enhanced combat feedback using screen shake, hit-stop frames, color flashing, and procedural scaling (blink/squash), ensuring the player always understands the combat state.

What i Learned:

Profiling for Web builds and managing 2D sprites within a 3D perspective. The importance of preloading shaders and particles to avoid compilation stutters on the web. How to implement data-driven systems to speed up content creation. Designing data-driven systems to accelerate content creation.

Developing Slime Dungeon solo reinforced the importance of prioritizing scalable architecture, even in short-term projects. By investing early in a robust FSM and a data-driven damage system, I was able to add new enemy types and power-ups within minutes during the final hours of the jam.
