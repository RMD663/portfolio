Slime Dungeon

Battle arena game with swarms of enemies on the screen. And stackable power ups.

About

Slime Dungeon is a 3D battle arena inspired by games such as Vampire Survivors, featuring hundreds of enemies on screen simultaneously. The project was originally developed in just three days for Mini Jam 202.

Main Issue

The jam version suffered from a critical performance issue: it was unable to maintain a stable framerate during the later stages of a run.

The primary cause was the way enemy physics were handled. Each enemy was implemented as a CharacterBody3D, which relies on an internal physics solver and per-frame callbacks. While suitable for small entity counts, this approach introduces a significant processing cost when scaled to hundreds or thousands of instances.

As a result, the game required a fundamental rework of how physics updates were processed.

Constraints

Because of time constraints the solution needed to improve performance without requiring a full rewrite of the existing architecture or the implementation of a custom physics body from scratch.

At peak moments, the game could have thousands of active entities, all executing physics updates every frame. This resulted in a severe performance curve, where CPU cost scaled linearly with the number of active enemies, causing major frame drops.

To address this, I analyzed some alternatives like time splicing or queuing, i opted for a staggered physics update system, distributing enemy physics processing across multiple frames instead of updating all entities simultaneously.

The Solution

The main solution was to built on top of the already existing Entities container, which was responsible for tracking active entities and handling global game state. Instead of allowing each enemy to process its own physics independently, the container was remade to work as a physics manager.

Enemy physics updates were removed from the individual entities and delegated to the container. This made it possible to fully control when and how many enemies would execute physics logic per frame.

How it Works

All active enemies are stored in a list managed by the Entities container. Rather than processing the entire list every frame, enemies are divided into fixed-size groups.

Each frame, only one group is processed. Enemies that are not updated during a given frame temporarily skip their physics update.

To compensate for the skipped frames, each processed enemy scales its velocity proportionally to the number of groups, ensuring that total movement over time remains consistent.

Implementation

The group size is defined upfront, and all active enemies are stored in a centralized list: 

 ## group vars
var splice_group_size : int = 50
var enemies : Array[Enemy] = []
var _counter : int = 0

Before processing, the system ensures the list is not empty and calculates how many groups are required. The number of processed entities per frame is clamped to avoid out-of-bounds access. 

func process_enemy_physics() -> void:
    var total : int = enemies.size()
    if total <= 0:
        return
    var total_groups : float = float(total) / float(splice_group_size)
    if total_groups < 1.0:
        total_groups = 1.0
    var iterations : int = min(splice_group_size, total)

Each frame on the container physics process, a group of enemies is processed. 

for i in range(iterations):
    if _counter >= total:
        _counter = 0
    if _counter >= enemies.size():
        _counter = 0
        return
    var enemy : Enemy = enemies[_counter]
    if is_instance_valid(enemy):
        var original_velocity : Vector3 = enemy.velocity
        enemy.velocity *= total_groups
        enemy.move_and_slide()
        enemy.velocity = original_velocity
        _counter += 1
    else:
        enemies.remove_at(_counter)

Trade-offs

This approach drastically improved performance by eliminating large CPU spikes caused by simultaneous physics updates.

Because enemies do not update their physics every physics tick, movement is evaluated at a lower temporal resolution. While velocity scaling compensates for skipped updates, intermediate collision checks and physical responses are not evaluated.

In practice, this results in:

  • Less precise collision responses between enemies
  • Occasional stiffness in enemy movement under dense clustering

For Slime Dungeon, these trade-offs were acceptable. Since the game prioritizes enemy density, visual clarity, and combat readability over per-entity physical accuracy. The staggered update system allowed the game to scale to hundreds of active enemies while preserving the overall feel and responsiveness of game-play. On other types of games this system may show itself as unreliable because of the lower collision accuracy and delay between checks.
