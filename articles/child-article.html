Child of Warana

Sacrifice is the only way up.

About

Child of Warana is my first attempt at a game jam, developed in just 3 days. The game's main mechanic is the 'Warana' eating: you can consume a Warana fruit to travel back in time, and a platform is created on the spot where you ate the fruit.

The Jam

During the design of CoW (Child of Warana), there were a handful of ideas of how the game would look and work. The jam theme was 'Dejavu'.

The first idea with this theme was a mountain climbing game where, after each attempt, you would create a ghost and needed to beat them climbing the mountain first. The second design was a platformer where the player would first go to place the platforms in the mountain, and after that, they would need to climb the entire mountain in one go.

The final design that actually entered production was the simplest: the player would have a collectible fruit, and if they ate the fruit, they would go back in time, leaving a platform behind.

The idea was defined, and I started coding the mechanic.

How I Implemented

To implement the time travel mechanic, I created a not-so-optimal way of doing it.

When the event for eating the fruit was triggered, the game would: get the player, initialize the animation, get his last checkpoint, delete the fruit from inventory, play the screen fade animation, show the platform created, play the fade out animation, and, at the end, teleport the player to the last checkpoint.

Even though this solution solved the main problem, it was absolute spaghetti to work with. The game is fully coupled: the inventory controls the player and the screen animation, and there are async functions that need tight timing to make things work. It was functional for the jam's purpose, but it was unmaintainable. If a change was made to the screen fade animation, the timing would break completely.

How I would implement now

If I had to recreate this system in another game or an eventual remake, I would follow another approach. In the original CoW, the system is tightly coupled in a handful of components.

The inventory does not need to know about the player nor the Player's Finite State Machine; this created a lot of problems. Instead, to control the transformation state, I would implement an FSM to manage game states, like start, pause, end... and, of course, transformation.

Once the game enters the transformation state, the game manager should handle things by itself, like starting the fade out and telling the player object to enter the transform state.

The flow would be something like:

1. Player ate the fruit

2. Emit "fruit eaten" signal

3. Game Manager receives the signal and enters 'transform' state

4. Emit "transformation started" signal

5. Listeners start the animation sequence

6. On finishing the transformation, emit "finished" signal

7. Game Manager receives the signal and enters 'resume' state

8. Game resumes and gives back player control

Results

In the end, even if the game did not have the best systems and architecture, it was still fun for some people. Because of that, I got 32nd out of 97 submissions on the jam. This means that the game can be fun, but still has some rough edges.

With the release of CoW, I learned that good planning is essential for a complex project, especially games. I intend to revisit the game in a future update or even a full remake.
